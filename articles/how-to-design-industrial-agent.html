引言：当 Agent 走出实验室

有了 vibe coding 的工具之后，做 Agent 似乎变得越来越容易：选一个大模型，写 Prompt，接上工具，再加上流程规划，代码交给 AI 来写。一个"智能体"demo 就诞生了，跑起来效果还不错。

但是当它接入真实业务场景时，当任务从单轮问答升级为多步决策，系统需要持续运行时，很多问题就会涌现出来：多步骤任务无法稳定复现，系统长期运行后状态逐渐失控。

Google 的《Agentic Design Patterns》给出了这个答案。我们应该如何设计一个真正可上线、可扩展、可治理的 Agent 架构？这本书最有价值的地方在于他是一个模式组合的思维，它把 Agent 架构拆成了 21 种模式，通过多个模式像拼图一样巧妙的组合，打造出适应各种复杂场景的智能体系统。

核心观点：非常建议每个 Agent 产品经理逐字阅读。Agent 时代，真正拉开产品经理差距的，恰恰就是 Agent 的架构设计能力。

① 控制流层：让任务真正跑起来

流程控制决定了智能体如何拆解和调度任务，是将概率性的语言模型转化为具备确定性工作流的核心架构。

Prompt Chaining（提示链）

就像流水线一样，把复杂任务拆解成一系列顺序执行的步骤。每一步的输出作为下一步的输入，通过指定结构化输出（如 JSON），确保数据在步骤间精确传递，提升了多步处理的可靠性。

示例：分析一篇文章：先提取关键信息，再做情感分析，最后生成摘要。

Routing（路由）

路由机制为智能体引入了条件逻辑，使智能体能根据输入上下文选择一个合适路径或工具。智能体系统突破了线性执行的限制，具备了上下文自适应能力。

Parallelization（并行化）

并行化允许智能体同时执行多个无依赖关系的子任务，并在后续节点汇总结果，大幅缩短整体执行时间，提升效率。

Planning（规划）

上面三种是人为的规划好路径，让智能体按照既定的路径走。而 Planning 是由智能体自己来规划路径，智能体不再是一个执行者而是成为一个战略规划者，它会先分析目标，制定计划，然后再执行。非常典型的需要规划的场景是做 deep research。

② 推理增强层：让智能体想得更清晰

这类模式解决的是 Agent 的"脑力"问题：如何让 LLM 想得更深入、推理更可靠。

思维链（CoT）

模型在生成最终答案之前，会先将问题拆解，逐步推导出 A、B、C 等中间步骤。通过显式推理，提升模型多步推理任务的表现。

思维树（ToT）

比思维链逻辑复杂度更进一步，思维链是单条思考路径，思维树是探索多个分支，评估每个分支，选择最优解。

ReAct

推理与行动交替进行。不是先想完再做，而是边想边做，循环的进行：思考-行动-观察。根据结果调整思路，更符合人类解决复杂问题的真实方式。

Reflection（反思）

智能体具备自省能力，生成结果后，自我反思，答案是否合理，逻辑是否严谨。Reflection 有两种方式，一种是智能体自己回顾反思，还有一种是创建一个独立的"批评者"，按照特定的标准进行更严格的改进。

这四种方式，思维链和思维树是前置思考，先想清楚再行动；ReAct 是过程中思考，边想边做；Reflection 是后置思考，做完了自省一下。

③ 状态与记忆层：让智能体记得住

人类有短期记忆和长期记忆，智能体也需要。智能体的记忆管理让它能够提供更个性化、更连贯的服务。这是从"一次性对话"到"持续关系"的关键。

短期记忆（上下文记忆）

类似于工作记忆，保存当前正在处理或最近访问的信息。包含最近的消息、智能体回复、工具使用结果以及当前交互中的智能体反思。这些内容共同影响 LLM 的后续响应和行为。但这些上下文依然是临时的，会在会话结束后丢失。

长期记忆（持久记忆）

智能体需要在多次交互、任务或较长周期内保留信息的仓库，类似于长期知识库。智能体需要长期记忆中的信息时，会查询外部存储，检索相关数据，并将其整合到短期上下文中，实现知识的融合。

④ 外部交互层：接入真实世界

智能体不能只活在"文字世界"里，它需要连接真实世界。

Tool Use（工具调用）

赋予智能体"手和脚"。它可以调用搜索引擎、计算器、数据库、API，从"只会说话"变成"能做事情"。

MCP（模型上下文协议）

这是一个标准化的连接协议，就像 USB 接口统一了各种设备。MCP 让智能体能够以标准化方式连接各种外部数据源，不需要为每个数据源单独开发适配器。

Tool Use vs MCP：工具函数调用就像给 AI 配一套专用工具（如特定扳手和螺丝刀），适合固定任务场景。而 MCP 则像通用电源插座系统，不直接提供工具，但允许任何合规工具接入，打造动态、可扩展的智能体工作坊。

RAG（检索增强生成）

智能体不可能记住所有知识，而且知识会不断更新。检索增强生成模式让 LLM 在生成响应前，先通过向量检索或混合检索从外部知识库中提取最相关的片段以丰富上下文，从根本上减少幻觉并确保回答具有可验证的事实依据。

Agentic RAG（智能体 RAG）

进阶的智能体 RAG 引入了推理层，智能体不仅被动接收检索结果，还会主动评估来源质量、调和信息冲突并进行多步推理。

⑤ 系统协作层：可分工、可扩展

多智能体协作（Multi-Agent Collaboration）

单一的智能体在面对复杂多领域任务的时候，能力往往受限。这个时候就需要将系统结构化为多个独立的智能体，进行团队协作。多个智能体的协作也有非常多的形式：

  • 顺序交接：一个智能体完成任务后，将输出传递给下一个智能体
  • 并行处理：多个智能体同时处理问题不同部分，结果后续合并
  • 辩论与共识：智能体基于不同视角和信息源讨论，最终达成共识或更优决策
  • 层级结构：管理者智能体根据工具或插件能力动态分配任务给工作智能体，并综合结果
  • 专家团队：不同领域专长智能体（如研究员、写作者、编辑）协作完成复杂输出
  • 批评-审查者：一个智能体生成初步输出内容，另一个智能体对其进行安全、合规的审查

智能体间通信（A2A Protocol）

每个智能体专注于自己擅长的领域，通过 A2A（智能体间通信）协议进行沟通协作。A2A 提供了一种基于 HTTP 的通用通信标准。通过发布包含能力与端点信息的"Agent Card"，智能体之间可以互相发现、委派异步任务并共享上下文，极大地促进了多智能体生态的协作。

⑥ 可靠性治理层：可控、可持续

一个好用的系统不仅仅是能够满足功能上的需求，它同时需要是稳定的、安全的。所以在做智能体的设计时，可靠性的治理也是一个非常重要的环节。

评估与监控（Evaluation & Monitoring）

传统的单元测试无法应对 LLM 的概率性特征。智能体运维需要持续追踪"智能体轨迹"，监控响应延迟与 Token 资源消耗，并采用"LLM 作为裁判（LLM-as-a-Judge）"模式，根据黄金标准自动化评估其响应的事实正确性与指令遵循度。

护栏与安全模式（Guardrails）

为防止智能体生成有害、越界内容或遭受提示词注入（Jailbreaking）攻击，必须实施多层防御。这包括输入内容的清洗、输出结果的拦截过滤，以及利用计算成本较低的快速模型对核心智能体的响应进行预筛查。

异常处理与恢复（Recovery）

解决智能体运行过程中出现故障的需求。包括错误日志记录、重试、备用方案、降级还包括恢复机制。

人类参与环节（Human-in-the-Loop, HITL）

在涉及高风险、伦理判断或高度模糊的场景中，完全自治是不可取的。HITL 模式将人类作为最终的验证者、策略指导者或纠错者，在关键节点必须触发 HITL，请求人类审核与批准，确保系统的安全与价值观对齐。

模式组合：1+1>2 的力量

不要孤立地看待每个模式，要学会组合！这才是设计模式的真正力量。

Prompt Chaining + Routing + RAG + Tools
= 智能客服系统
Planning + ReAct + Tools
= 研究助手
Planning + Multi-Agent + Reflection + Memory
= 能够自主完成复杂项目的 AI 团队
HITL + Guardrails + Recovery
= 安全可控的自动化决策系统

实践建议

那么，如何在实际项目中应用这些设计模式呢？我给大家五个实践建议：

1
从简单开始
不要一开始就构建复杂的多智能体系统，先从提示链、工具调用等基础模式开始，逐步增加复杂度。就像登山，要一步步往上走。
2
根据场景选择
没有"最好"的模式组合，只有"最适合"的。分析你的具体需求，选择合适的模式。
3
持续迭代
智能体系统的优化是一个持续过程。先构建 MVP，然后根据实际运行情况不断优化。
4
重视可靠性
不要只关注功能，安全、监控、异常处理同样重要。一个不可控的强大系统比一个可控的简单系统更危险。
5
拥抱开源生态
LangChain、Google ADK、CrewAI 等框架已经实现了大部分设计模式，不需要从零开始造轮子。

结语：架构设计是核心竞争力

Google 的《Agentic Design Patterns》给我们最大的启示是：Agent 系统的竞争力不在于单点技术的先进性，而在于架构设计的合理性。

21 种设计模式就像乐高积木，如何组合、如何取舍、如何平衡复杂度与效果，考验的是产品经理的架构思维。

在 Agent 时代，能够设计出可上线、可扩展、可治理的智能体系统的产品经理，将成为最稀缺的人才。
