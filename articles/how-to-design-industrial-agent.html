引言：当 Agent 走出实验室

2025 年，AI Agent 已经从概念验证走向了生产环境。但当我们在真实业务场景中部署 Agent 时，一个残酷的现实浮现出来：Demo 里跑得通的智能体，在持续运行、多步决策、复杂交互的场景下，往往会暴露出稳定性差、状态失控、难以扩展等问题。

Google 发布的《Agentic Design Patterns》白皮书（400+ 页）正是为了解决这些问题。这份文档的价值不在于它罗列了多少种模式，而在于它提供了一种组合式思维——将 Agent 架构拆解为 21 种可复用的设计模式，通过像拼图一样巧妙组合，打造出适应各种复杂场景的工业级智能体系统。

核心观点：对于 Agent 产品经理而言，这本书值得逐字阅读。因为在 Agent 时代，真正拉开产品经理差距的，恰恰是架构设计能力。

一、六大类别：Agent 架构的全景图

Google 将这 21 种模式按照功能划分为六大类别，每一类解决 Agent 系统中的一个核心问题：

类别 核心解决的问题 关键模式
流程控制 智能体如何把任务跑起来 Prompt Chaining、Routing、Parallelization、Planning
推理提升 智能体怎么想得更清晰、结果更可靠 CoT、ToT、ReAct、Reflection
外部交互 智能体怎么连接真实世界 Tool Use、MCP、RAG
记忆管理 保持智能体内部连贯性 短期记忆、长期记忆
协作 提升智能体解决复杂问题的能力 多智能体协作、A2A 协议
治理 确保系统安全可控 评估监控、护栏、异常处理、HITL

这六大类别构成了一个完整的 Agent 系统架构闭环：从任务启动到推理执行，从外部交互到记忆保持，从团队协作到安全治理。

二、流程控制：让任务有序运行

流程控制类模式解决的是最基础的问题：Agent 如何组织和执行任务流程。

2.1 Prompt Chaining（提示链）

核心思想：像流水线一样，将复杂任务拆解成一系列顺序执行的步骤。

步骤 1：提取文章关键信息
↓
步骤 2：进行情感分析
↓
步骤 3：生成内容摘要

MVP 阶段 低成本 固定流程

2.2 Routing（路由）

核心思想：引入条件逻辑，使智能体能根据输入上下文选择合适的路径或工具。

用户问题 → 意图识别
↓
技术问题
→ 技术支持 Agent
账单问题
→ 财务处理 Agent
投诉建议
→ 客户关系 Agent
其他问题
→ 通用客服 Agent

与 Prompt Chaining 的区别：Routing 是"分支选择"，Prompt Chaining 是"顺序执行"。

2.3 Parallelization（并行化）

核心思想：将不相干的独立子任务同时运行，提升整体效率。

用户请求：生成季度数据分析报告
↓
并行执行
子任务 A
统计用户行为
3s
子任务 B
分析市场趋势
4s
子任务 C
整理竞品信息
3s
↓
汇总结果 → 生成报告

2.4 Planning（规划）

核心思想：由智能体自主规划执行路径，从"执行者"升级为"战略规划者"。

用户输入：帮我研究一下新能源汽车市场
↓ Agent 自主规划
  1. 搜索市场规模数据
  2. 分析主要玩家（特斯拉、比亚迪等）
  3. 查找技术趋势
  4. 整理政策影响
  5. 生成综合分析报告
↓ 执行过程中动态调整计划

关键区别：前三种模式是人为规划好路径，Planning 是智能体自己规划路径。

三、推理提升：让思考更清晰

推理提升类模式解决的是 Agent 的"脑力"问题：如何让 LLM 想得更深入、推理更可靠。

3.1 思维链（Chain of Thought, CoT）

核心思想：模型在生成最终答案前，先将问题拆解，逐步推导出中间步骤。

问题：一个农场有 35 只鸡和 20 只兔子，一共有多少条腿？
推理过程：
• 每只鸡有 2 条腿，35 只鸡有 35 × 2 = 70 条腿
• 每只兔子有 4 条腿，20 只兔子有 20 × 4 = 80 条腿
• 总共：70 + 80 = 150 条腿

答案：150 条腿

3.2 思维树（Tree of Thought, ToT）

核心思想：比 CoT 更进一步，探索多个思考分支，评估每个分支，选择最优解。

对比维度CoTToT
思考路径单条路径（线性）多分支探索（树状）
适用场景数学计算、逻辑推理创意写作、策略规划
复杂度低高

3.3 ReAct（推理+行动）

核心思想：推理与行动交替进行，不是"先想完再做"，而是"边想边做"。

思考 Thought
→
行动 Action
→
观察 Observation
→ ...

3.4 Reflection（反思）

核心思想：智能体具备自省能力，生成结果后自我反思答案是否合理、逻辑是否严谨。

方式一
自我反思
智能体自己回顾检查
方式二
独立批评者
创建独立"批评者"智能体，按特定标准严格审查

四种推理模式对比

模式思考时机特点
CoT前置思考先想清楚再行动
ToT前置思考多路径探索
ReAct过程中思考边想边做
Reflection后置思考做完了自省

四、外部交互：连接真实世界

智能体不能只活在"文字世界"里，它需要连接真实世界获取数据、执行操作。

4.1 Tool Use（工具调用）

核心思想：赋予智能体"手和脚"，让它能调用搜索引擎、计算器、数据库、API 等外部工具。

这是从"只会说话"到"能做事情"的关键一跃。

用户："帮我查一下订单 12345 的物流"
↓
LLM 判断：需要调用"查询物流"工具
↓
系统调用：get_logistics(order_id="12345")
↓
返回结果：{status: "已发货", courier: "顺丰"}
↓
LLM 整理回复用户

4.2 MCP（模型上下文协议）

核心思想：标准化的连接协议，让智能体以统一方式连接各种外部数据源。

对比Tool UseMCP
类比专用工具（特定扳手）通用接口（USB/插座）
适用固定任务场景动态、可扩展场景
开发每个工具单独适配标准化接入

4.3 RAG（检索增强生成）

核心思想：智能体不可能记住所有知识，且知识会不断更新。RAG 让它能从外部知识库检索信息，然后基于检索内容生成回答。

用户提问："iPhone 15 怎么退货？"
↓
向量化检索 → 找到相关文档
↓
获取文档："iPhone 15 退货政策.pdf"
↓
结合上下文生成回答
↓
输出："iPhone 15 支持7天无理由退货..."

五、记忆管理：从一次性对话到持续关系

人类有短期记忆和长期记忆，智能体也需要。记忆管理是 Agent 从"一次性对话"进化到"持续关系"的关键。

5.1 短期记忆（上下文记忆）

作用：类似于工作记忆，保存当前正在处理或最近访问的信息。

1
最近的消息历史
2
智能体的回复
3
工具使用结果
4
当前交互中的反思内容
特点：影响 LLM 后续响应，但会话结束后丢失

5.2 长期记忆（持久记忆）

作用：在多次交互、任务或较长周期内保留信息的仓库，类似于长期知识库。

智能体需要信息时
↓
查询外部存储（向量数据库等）
↓
检索相关数据
↓
整合到短期上下文
↓
实现知识融合

六、系统协作：多智能体团队

单一的智能体在面对复杂多领域任务时，能力往往受限。这时候就需要将系统结构化为多个独立的智能体，进行团队协作。

6.1 协作形式

A
专家团队模式
研究员 Agent：负责信息搜集和研究
编辑 Agent：负责内容整理和润色
资料收集 Agent：负责数据获取和整理
B
对抗模式
内容生成 Agent：负责创作内容
内容审查 Agent：负责审核和提出改进意见

6.2 A2A 协议（Agent-to-Agent）

核心作用：定义智能体之间的通信标准，让不同智能体能够相互理解、协作配合。

价值：让专业化的智能体各司其职，协同完成复杂任务。

七、可靠性与治理：让系统安全可控

一个好用的系统不仅仅是满足功能需求，它同时需要是稳定的、安全的、可控的。

7.1 评估与监控

评估智能体的轨迹路径、准确率、延迟、资源消耗（LLM Token 用量等），确保智能体始终符合预期目标。

7.2 护栏与安全（Guardrails）

作为保护层，引导智能体的行为，确保输出可信、可预测、有益。

1
内容安全过滤
防止生成有害内容
2
输出格式校验
确保输出符合预期格式
3
敏感信息保护
防止泄露隐私数据

7.3 异常处理与恢复

机制包括：错误日志记录、自动重试机制、备用方案切换、服务降级策略、状态恢复机制。

7.4 Human-in-the-Loop（人机协同）

在需要细致判断、伦理推理或复杂模糊语境理解的任务上，引入人类监督，确保 AI 行为符合人类价值和安全规范。

八、模式组合：1+1>2 的力量

最重要的认知：不要孤立地看待每个模式，要学会组合！这才是设计模式的真正力量。

8.1 经典组合示例

组合能力描述应用场景
提示链 + 工具调用 + RAG能够联网查资料并逐步分析研究助手
规划 + 多智能体协作 + 反思 + 记忆管理能够自主完成复杂项目AI 项目管理团队
路由 + 并行化 + 资源感知优化高效且经济的任务处理智能客服系统
HITL + 护栏 + 异常处理安全可控的自动化决策金融风控、医疗诊断

九、实践建议：从理论到落地

1
从简单开始
不要一开始就构建复杂的多智能体系统。先从提示链、工具调用等基础模式开始，逐步增加复杂度。就像登山，要一步步往上走。
2
根据场景选择
没有"最好"的模式组合，只有"最适合"的。分析你的具体需求，选择合适的模式。
3
持续迭代
智能体系统的优化是一个持续过程。先构建 MVP，然后根据实际运行情况不断优化。
4
重视可靠性
不要只关注功能，安全、监控、异常处理同样重要。
5
拥抱开源生态
LangChain、Google ADK、CrewAI 等框架已经实现了大部分设计模式，不需要从零开始造轮子。
一个不可控的强大系统，比一个可控的简单系统更危险。

结语：架构设计是核心竞争力

Google 的《Agentic Design Patterns》给我们最大的启示是：Agent 系统的竞争力不在于单点技术的先进性，而在于架构设计的合理性。

21 种设计模式就像乐高积木，如何组合、如何取舍、如何平衡复杂度与效果，考验的是产品经理的架构思维。

在 Agent 时代，能够设计出可上线、可扩展、可治理的智能体系统的产品经理，将成为最稀缺的人才。
