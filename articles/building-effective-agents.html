构建高效 Agent：Anthropic 实战经验总结

从数十个跨行业团队的实践中提炼的 5 大工作流模式，以及何时该用 Workflow、何时该用 Agent 的决策框架。

引言

过去一年，Anthropic 与数十个跨行业团队合作构建大语言模型（LLM）Agent。一个令人惊讶的发现是：最成功的实现往往不是使用复杂框架或专用库，而是采用简单、可组合的模式。

在这篇文章中，我们将分享从客户合作和自身实践中总结的经验，为开发者提供构建高效 Agent 的实用建议。文章基于 Anthropic 的研究报告《Building Effective Agents》，并结合实际应用场景进行解读。

核心观点：构建 Agent 的成功不在于构建最复杂的系统，而在于构建最适合你需求的系统。从简单提示开始，通过全面评估进行优化，只有当简单方案无法满足需求时才添加多步骤的 Agent 系统。

什么是 Agent？

"Agent" 这个词有多种定义。一些团队将其定义为完全自主的系统，能够在较长时间内独立运行，使用各种工具完成复杂任务。另一些团队则用这个词来描述遵循预定义工作流的更规范化的实现。

在 Anthropic 的分类体系中，所有这些变体都属于 Agentic Systems（类 Agent 系统），但在架构上有一个重要区分：

🔀 Workflow（工作流）

LLM 和工具通过预定义的代码路径进行编排。系统按照固定的步骤执行，每一步都是确定的。

🤖 Agent（智能体）

LLM 动态地指导自己的流程和工具使用，保持对如何完成任务的控制权。系统能够根据情况自主决策。

何时使用（或不使用）Agent

在使用 LLM 构建应用时，我们建议寻找最简单的解决方案，只有在必要时才增加复杂度。这可能意味着完全不构建 Agent 系统。

简单优先原则：Agent 系统通常用延迟和成本换取更好的任务性能。在考虑这种权衡是否有意义之前，先尝试优化单次 LLM 调用，通过检索和上下文示例通常就足够了。

三种选择层级：

  • 单次 LLM 调用：适用于简单任务，通过检索增强和上下文学习即可满足需求
  • Workflow：适用于定义明确的任务，需要可预测性和一致性
  • Agent：适用于需要灵活性和模型驱动决策的开放式问题

构建块：增强型 LLM

Agentic 系统的基础构建块是一个通过增强功能（如检索、工具和记忆）强化的 LLM。当前模型能够主动使用这些能力——生成自己的搜索查询、选择合适的工具，并决定保留哪些信息。

增强型 LLM：基础构建块，具备检索、工具和记忆能力

实现这些增强功能的关键在于两个方面：

  • 根据你的具体用例定制这些能力
  • 确保它们为 LLM 提供简单、文档完善的接口

Anthropic 最近发布的 Model Context Protocol (MCP) 就是一种实现方式，它允许开发者通过简单的客户端实现与不断增长的第三方工具生态系统集成。

Workflow 工作流模式

接下来，我们将探索生产中常见的 Agentic 系统模式。从基础构建块开始，逐步增加复杂度，从简单的组合工作流到自主 Agent。

1. Prompt Chaining（提示链）

提示链将任务分解为一系列步骤，每个 LLM 调用处理前一个调用的输出。你可以在任意中间步骤添加程序化检查（见下图中的"gate"）以确保流程正常进行。

提示链工作流：将任务分解为顺序步骤

适用场景：任务可以轻松清晰地分解为固定子任务的情况。主要目标是通过让每个 LLM 调用处理更简单的任务，用延迟换取更高的准确性。

应用示例：

  • 生成营销文案，然后将其翻译成不同语言
  • 编写文档大纲，检查大纲是否符合特定标准，然后基于大纲编写文档

2. Routing（路由）

路由对输入进行分类，并将其引导至专门的后续任务。这种工作流允许关注点分离，并构建更专门的提示。如果没有这种工作流，针对一种输入的优化可能会损害其他输入的性能。

路由工作流：将输入分类并引导至专门处理流程

适用场景：复杂任务中存在不同的类别，分别处理效果更好，且分类可以由 LLM 或更传统的分类模型/算法准确处理的情况。

应用示例：

  • 将不同类型的客户服务查询（一般问题、退款请求、技术支持）引导至不同的下游流程、提示和工具
  • 将简单/常见问题路由到更小、成本效益更高的模型（如 Claude Haiku），将困难/异常问题路由到能力更强的模型（如 Claude Sonnet）

3. Parallelization（并行化）

LLM 有时可以同时处理任务，并通过编程方式聚合输出。这种工作流有两种关键变体：

  • 分段（Sectioning）：将任务分解为独立的子任务并行运行
  • 投票（Voting）：多次运行同一任务以获得多样化的输出

并行化工作流：同时处理任务并聚合结果

适用场景：当分解的子任务可以并行化以提高速度，或者需要多个视角或尝试以获得更高置信度结果时。对于具有多个考虑因素的复杂任务，当每个考虑因素由单独的 LLM 调用处理时，LLM 通常表现更好，允许专注于每个特定方面。

应用示例：

  • 分段：实现防护栏，其中一个模型实例处理用户查询，另一个筛选不当内容。这比让同一个 LLM 调用同时处理防护栏和核心响应表现更好
  • 分段：自动化评估 LLM 性能，每个 LLM 调用评估模型在给定提示上的性能的不同方面
  • 投票：审查代码漏洞，多个不同的提示审查代码并在发现问题时标记
  • 投票：评估给定内容是否不当，多个提示评估不同方面或需要不同的投票阈值来平衡误报和漏报

4. Orchestrator-Workers（编排器-工作者）

在编排器-工作者工作流中，一个中心 LLM 动态分解任务，将其委派给工作者 LLM，并综合它们的结果。

编排器-工作者工作流：中心节点动态分解和委派任务

适用场景：适用于无法预测所需子任务的复杂任务（例如，在编码中，需要更改的文件数量和每个文件中更改的性质可能取决于任务）。虽然与并行化在拓扑上相似，但关键区别在于其灵活性——子任务不是预定义的，而是由编排器根据特定输入确定的。

应用示例：

  • 编码产品，每次对多个文件进行复杂更改
  • 搜索任务，涉及从多个来源收集和分析信息以寻找可能相关的信息

5. Evaluator-Optimizer（评估器-优化器）

在评估器-优化器工作流中，一个 LLM 调用生成响应，另一个在循环中提供评估和反馈。

评估器-优化器工作流：迭代改进响应质量

适用场景：当我们有明确的评估标准，且迭代改进能够提供可衡量的价值时，这种工作流特别有效。两个适合的迹象是：第一，当人类表达反馈时，LLM 响应可以明显改进；第二，LLM 能够提供这样的反馈。这类似于人类作家在撰写润色文档时可能经历的迭代写作过程。

应用示例：

  • 文学翻译，其中存在翻译 LLM 最初可能无法捕捉的细微差别，但评估器 LLM 可以提供有用的批评
  • 复杂搜索任务，需要多轮搜索和分析来收集全面信息，评估器决定是否需要进一步搜索

自主 Agent

随着 LLM 在关键能力上的成熟——理解复杂输入、进行推理和规划、可靠地使用工具以及从错误中恢复——Agent 正在生产中涌现。

Agent 从人类用户的命令或交互式讨论开始工作。一旦任务明确，Agent 就会独立规划和运行，可能在需要进一步信息或判断时返回给人类。在执行过程中，Agent 必须在每个步骤从环境中获得"基本事实"（如工具调用结果或代码执行）以评估其进展。Agent 可以在检查点或遇到障碍时暂停以获取人类反馈。

自主 Agent：基于环境反馈循环使用工具

任务通常在完成后终止，但包含停止条件（如最大迭代次数）以保持控制也很常见。

适用场景：Agent 可用于开放式问题，其中难以或不可能预测所需步骤数，且无法硬编码固定路径。LLM 可能会运行多轮，你必须对其决策有一定程度的信任。Agent 的自主性使其成为在受信任环境中扩展任务的理想选择。

⚠️ 注意事项：Agent 的自主性意味着更高的成本和潜在的复合错误。我们建议在沙盒环境中进行广泛测试，并设置适当的防护栏。

应用示例：

  • 编码 Agent 解决 SWE-bench 任务，涉及根据任务描述对多个文件进行编辑
  • "计算机使用"参考实现，Claude 使用计算机完成任务

总结与最佳实践

这些构建块不是规定性的。它们是开发者可以根据不同用例塑造和组合的常见模式。成功的关键与任何 LLM 功能一样：衡量性能并迭代实现。重申：你应该只在复杂度能够明显改善结果时才考虑添加复杂度。

在实现 Agent 时，我们尝试遵循三个核心原则：

1️⃣ 保持 Agent 设计的简单性

不要为了复杂而复杂。每个增加的组件都应该有明确的理由。

2️⃣ 通过明确展示 Agent 的规划步骤来优先考虑透明性

让用户了解 Agent 正在做什么、为什么要这样做。可解释性建立信任。

3️⃣ 通过彻底的工具文档和测试精心打造 Agent-计算机接口（ACI）

工具是 Agent 与外部世界交互的桥梁，其设计质量直接影响 Agent 的可靠性。

框架可以帮助你快速入门，但在进入生产环境时，不要犹豫减少抽象层并使用基本组件构建。通过遵循这些原则，你可以创建不仅强大而且可靠、可维护且受用户信任的 Agent。

最后的建议：从直接使用 LLM API 开始——许多模式可以用几行代码实现。如果你确实使用框架，确保你理解底层代码。对底层内容的错误假设是客户错误的常见来源。
