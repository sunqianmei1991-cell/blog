核心观点：模式组合的力量

Google 在《Building Effective Agents》白皮书中给出了来自数十个跨行业团队的实践总结：不要孤立地使用单一模式，真正的威力在于多种模式的巧妙组合。

关键洞察

从 Anthropic、Google、OpenAI 等公司的实践来看，90% 以上的生产级 Agent 都采用了Workflow（预定义工作流）模式，而非完全自主的Agent（自主规划）模式。原因在于：Workflow 更可控、更可预测、更容易优化。

这篇文章将深入解析 5 大核心工作流模式，以及何时该用 Workflow、何时该用 Agent 的决策框架。

5 大核心工作流模式

Google 将 Agent 的实现模式归纳为 5 种核心工作流，这是从数十个跨行业真实项目中提炼的最佳实践：

Prompt Chaining
将复杂任务拆解为顺序执行的多个步骤，每一步的输出作为下一步的输入
Parallelization
并行执行多个独立子任务，然后汇总结果，提升效率
Routing
根据输入类型或特征，路由到不同的专用处理流程
Orchestrator-Workers
中心协调者动态规划任务分配，多个工作单元并行执行
Evaluator-Optimizer
迭代评估与优化，直到结果达到预期标准

模式一：Prompt Chaining（提示链）

核心理念

将复杂任务拆解为一系列顺序执行的子任务，每个子任务由一个专门的 LLM 处理。前一个子任务的输出作为下一个子任务的输入，形成流水线式的处理流程。

用户输入
↓
LLM 1
→
提取关键信息
↓
LLM 2
→
情感分析
↓
LLM 3
→
生成摘要
↓
最终输出

适用场景

  • 多步骤处理流程：如内容审核、报告生成、数据转换
  • 需要结构化输出：每一步需要严格的 JSON/Schema 输出
  • 任务可分解：每个子任务可以独立处理

工程实践

# Prompt Chaining 示例：文章分析流水线
from langgraph.graph import StateGraph
from typing import TypedDict

class ArticleState(TypedDict):
    raw_text: str
    key_info: dict
    sentiment: str
    summary: str

# Step 1: 提取关键信息
def extract_info(state: ArticleState) -> ArticleState:
    prompt = f"从以下文章中提取关键信息：{state['raw_text']}"
    state['key_info'] = llm.invoke(prompt)
    return state

# Step 2: 情感分析
def analyze_sentiment(state: ArticleState) -> ArticleState:
    prompt = f"分析以下内容的情感：{state['key_info']}"
    state['sentiment'] = llm.invoke(prompt)
    return state

# Step 3: 生成摘要
def generate_summary(state: ArticleState) -> ArticleState:
    prompt = f"基于以下信息生成摘要：{state['key_info']}, 情感：{state['sentiment']}"
    state['summary'] = llm.invoke(prompt)
    return state

# 构建工作流
workflow = StateGraph(ArticleState)
workflow.add_node("extract", extract_info)
workflow.add_node("analyze", analyze_sentiment)
workflow.add_node("summarize", generate_summary)
workflow.set_entry_point("extract")
workflow.add_edge("extract", "analyze")
workflow.add_edge("analyze", "summarize")
💡 最佳实践

每个步骤使用结构化输出（JSON Schema）确保数据在步骤间精确传递。建议为每个步骤添加错误处理和重试逻辑。

模式二：Parallelization（并行化）

核心理念

当任务包含多个相互独立的子任务时，可以同时执行这些子任务，最后汇总结果。这就像从"串行工厂"升级为"并行车间"，大幅提升处理效率。

用户输入
↓
LLM 1
→
搜索结果
|
LLM 2
→
数据库查询
|
LLM 3
→
API 调用
↓
结果聚合
→
最终响应

适用场景

  • 多源信息收集：如同时查询多个搜索引擎或数据库
  • 批量内容生成：如同时生成多个产品描述或报告章节
  • 多角度分析：如同时从不同维度分析同一问题

工程实践

# Parallelization 示例：并发执行多个任务
import asyncio
from concurrent.futures import ThreadPoolExecutor

async def parallel_search(query: str):
    # 并行执行多个搜索
    tasks = [
        search_google(query),
        search_baidu(query),
        search_github(query),
    ]
    results = await asyncio.gather(*tasks)

    # 汇总结果
    aggregated = aggregate_results(results)
    return aggregated

# 使用 ThreadPoolExecutor 处理同步任务
def batch_generate(prompts: List[str]):
    with ThreadPoolExecutor(max_workers=10) as executor:
        results = list(executor.map(llm.invoke, prompts))
    return results

模式三：Routing（路由）

核心理念

根据输入的特征（意图、复杂度、内容类型），将其分配到不同的处理路径。这不是简单的 if-else，而是基于 LLM 或向量相似度的智能路由。

用户查询
↓
意图分类器
↓
客服处理
|
技术问答
|
投诉处理
|
转人工

路由策略

策略 适用场景 优点
基于规则 简单分类、明确规则 快速、可控、无需 LLM
LLM 分类 复杂意图、多样化表达 灵活、适应性强
向量相似度 知识库匹配、FAQ 路由 语义匹配、零样本
成本路由 资源优化场景 成本控制、效率优化

工程实践

# Routing 示例：智能路由
class SmartRouter:
    def __init__(self):
        self.routes = {
            "technical": technical_agent,
            "sales": sales_agent,
            "support": support_agent,
        }

    def route(self, query: str) -> Agent:
        # 使用 LLM 进行意图分类
        intent = llm.invoke(
            f"分类以下查询的技术类型：{query}"
            + "返回：technical/sales/support"
        )

        # 成本感知路由：简单问题用小模型
        if is_simple_query(query):
            return cheap_agent  # 如 GPT-4o-mini

        return self.routes.get(intent, default_agent)

模式四：Orchestrator-Workers（编排器-工作器）

核心理念

这是最灵活的模式。一个中心"编排器"负责理解任务、规划步骤、动态分配子任务；多个"工作器"各自执行具体任务，最后由编排器汇总结果。

复杂任务
↓
编排器
↓
规划步骤
↓
工作器 A
||
工作器 B
||
工作器 C
↓
结果聚合
→
最终输出

适用场景

  • 复杂多步骤任务：如旅行规划、代码审查、多维度报告生成
  • 动态任务分配：任务步骤事前无法完全确定
  • 需要适应性：根据中间结果调整后续计划
与 Prompt Chaining 的区别

Prompt Chaining 是静态的，步骤顺序在运行前就已确定；Orchestrator-Workers 是动态的，编排器会根据任务进展实时决定下一步调用哪个工作器。

模式五：Evaluator-Optimizer（评估-优化）

核心理念

让 LLM 生成初始结果，然后让另一个 LLM（或同一 LLM 的不同实例）评估结果质量，如果不满意则进行优化，形成迭代循环。

初始输入
↓
生成器
→
候选输出
↓
评估器
↓
质量判断
↓
满意？
↓
✓ 是
✗ 否 → 优化

适用场景

  • 需要高质量输出：如代码生成、内容创作、报告撰写
  • 有明确的评估标准：如格式要求、完整性检查
  • 迭代优化有价值：如翻译润色、创意写作
# Evaluator-Optimizer 示例：代码生成与优化
async def generate_and_optimize(task: str, criteria: str):
    max_iterations = 3

    for i in range(max_iterations):
        # Step 1: 生成
        code = llm.generate(task)

        # Step 2: 评估
        evaluation = llm.evaluate(
            f"检查以下代码是否符合标准：{criteria}\n\n{code}"
        )

        # Step 3: 检查是否通过
        if evaluation.passed:
            return code

        # Step 4: 优化（提供反馈）
        feedback = evaluation.feedback
        task = f"基于以下反馈优化代码：{feedback}\n\n{task}"

    return code  # 返回最后一次生成的结果

决策框架：何时用 Workflow？何时用 Agent？

这是产品设计中最关键的问题。Google 给出了一个清晰的决策框架：

维度 Workflow（工作流） Agent（自主智能体）
任务复杂度 可预先定义步骤 步骤无法预先确定
可预测性 需要确定性的输出 可以接受探索性结果
错误容忍 错误代价高 错误可以接受
成本控制 需要精确控制成本 可以接受更高 token 消耗
调试需求 需要可追溯的执行 灵活性优先于可追溯
📌 实践建议

从 Workflow 开始：先尝试用预定义工作流解决问题，只有当任务真正需要动态规划、步骤无法预先定义时，才考虑使用 Agent。大多数场景下，Workflow + 好的 Prompt 就足够了。

进阶模式：Agentic RAG

RAG（检索增强生成）是让 LLM 接入私有知识库的核心技术。但基础 RAG 有局限性——Agentic RAG 引入了智能体思维，让检索变得更智能。

基础 RAG vs Agentic RAG

特性 基础 RAG Agentic RAG
检索策略 一次性向量检索 动态决定检索策略
结果评估 被动接收结果 主动评估相关性
错误处理 无法处理检索失败 可以尝试替代方案
多步推理 单次检索 多步迭代检索

Agentic RAG 工作流

用户问题
↓
理解问题
↓
生成检索查询
↓
向量检索
↓
评估相关性
↓
足够？
↓
✓ 生成答案
✗ 再次检索 
# Agentic RAG 示例
async def agentic_rag(query: str, max_retries: int = 3):
    # Step 1: 理解问题，制定检索策略
    search_plan = llm.plan_search(query)

    for attempt in range(max_retries):
        # Step 2: 执行检索
        results = await search_with_plan(search_plan, query)

        # Step 3: 评估结果质量
        relevance = llm.evaluate(
            f"评估以下结果与问题的相关性：\n问题：{query}\n结果：{results}"
        )

        if relevance.score > 0.8:
            break

        # Step 4: 如果不相关，改进检索策略
        search_plan = llm.refine_search(query, relevance.feedback)

    # Step 5: 生成答案
    answer = llm.generate(query, results)
    return answer

Tool Use：赋予 Agent 行动能力

Tool Use（工具使用）是 Agent 与外部世界交互的基础。LLM 通过生成结构化调用（如 JSON），让智能体执行搜索、计算、API 调用等操作。

Tool Use 架构

User Query
"帮我查下北京天气"
→
LLM
推理 + 生成调用
→
Tool Call
{"tool": "weather", "args": {"city": "北京"}}
→
Execution
API 返回天气数据

Tool 定义的最佳实践

# 好的 Tool 定义示例
weather_tool = {
    "name": "get_weather",
    "description": "获取指定城市的天气信息",
    "parameters": {
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "城市名称，如北京、上海",
            },
            "date": {
                "type": "string",
                "description": "日期，格式YYYY-MM-DD，默认今天",
            }
        },
        "required": ["city"]
    }
}
Tool 定义要点
  • description 要清晰描述工具用途，帮助 LLM 理解何时该调用
  • parameters 要有明确的类型和描述
  • 必填参数 要在 required 中声明

Evaluation：构建可量化的 Agent 系统

传统软件开发中的"测试"概念在 Agent 场景下不再适用——LLM 的输出是概率性的，需要新的评估范式。

评估维度

准确性
回答是否正确？事实是否有误？
指令遵循
是否按照用户要求的方式回答？
相关性
回答是否切中问题要点？
安全性
是否有有害内容？是否越界？

LLM-as-a-Judge 模式

使用 LLM 来评估 LLM 的输出，这是一种成本效益较高的自动化评估方式。

# LLM-as-a-Judge 示例
def evaluate_response(question: str, response: str, criteria: str):
    judge_prompt = f"""评估以下回答的质量。

问题：{question}
回答：{response}

评估标准：{criteria}

请给出：
1. 得分 (0-10)
2. 具体反馈
3. 改进建议
"""

    evaluation = llm.invoke(judge_prompt)
    return evaluation

可观测性：追踪 Agent 轨迹

🔍 需要追踪的关键指标
  • 执行轨迹：完整的思考-行动-观察日志
  • Token 消耗：每个请求的输入/输出 token 数
  • 延迟分布：P50/P95/P99 响应时间
  • 工具调用：工具使用成功率、错误类型分布

Guardrails：守护 Agent 的安全边界

Guardrails（护栏）是防止 Agent 出错、越界或被攻击的多层防御机制。

多层防御架构

输入层

输入清洗 · 恶意检测 · 格式验证

↓
处理层

Prompt 注入检测 · 内容过滤

↓
输出层

有害内容拦截 · 敏感信息脱敏

Guardrails 实践

# Guardrails 示例：输入验证
from guardrails import Guard

guard = Guard().from_string(
    """## 输入验证
    - 输入长度不能超过 2000 字符
    - 禁止包含恶意代码或脚本
    - 禁止政治敏感内容
    ## 输出格式
    - 回答必须是有效的 JSON 或纯文本
    - 禁止泄露敏感信息
    """
)

# 使用 Guardrails
validated_input = guard.validate(user_input)

结语：从模式到实践

Google 的《Building Effective Agents》给了我们最重要的启示：构建高效的 Agent 系统，关键不在于使用最先进的模型，而在于正确选择和组合设计模式。

从简单开始
先用 Prompt Chaining 和 Routing 解决问题，只有必要时才引入更复杂的模式。
重视评估和护栏
在构建之初就考虑可观测性和安全边界，这是生产环境的基础。
持续迭代
基于实际运行数据不断优化，Agent 系统是"炼"出来的，不是"设计"出来的。
