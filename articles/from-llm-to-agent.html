从 LLM 到 Agent：建立 AI Agent 认知框架

引言

对于刚接触 AI Agent 的产品经理而言，技术栈中的概念体系往往显得庞杂而模糊：LLM、Prompt Engineering、RAG、Agent、Context Management、Memory、Orchestration……这些术语各自指代什么？它们之间存在怎样的逻辑关联？

本文以智能客服系统为场景载体，从技术演进视角梳理从 LLM 到 Agent 的完整路径，帮助读者建立系统性的技术认知框架。

阅读本文后，你将获得
  • LLM 与 Agent 的本质差异及技术边界
  • Agent 系统的核心组件及其协作机制
  • Agent 技术选型的决策框架
  • Agent 在真实业务场景中的落地路径

一、LLM 的局限性及增强方案

1.1 LLM 的技术边界

LLM（Large Language Model）基于大规模语料预训练，具备强大的语言理解与生成能力。然而，当直接应用于企业级客服场景时，会暴露三个核心局限：

  • 知识边界：训练数据存在时间截断，无法获取实时信息
  • 领域适配：缺乏企业私有知识，回答偏离业务实际
  • 上下文限制：多轮对话中容易丢失历史信息

1.2 Prompt Engineering：角色定义与行为约束

Prompt Engineering 通过结构化指令为 LLM 设定角色边界与行为准则，是解决领域适配问题的首要手段。

system-prompt.txt 
[角色定义]
你是某电商平台的智能客服助手，负责处理用户咨询、订单查询、售后申请等业务。

[知识边界]
- 退货政策：7天无理由退货，商品需保持完好
- 运费规则：订单满99元包邮
- 会员权益：会员享受95折优惠

[行为约束]
1. 使用礼貌用语，称呼用户为"您"
2. 对不确定信息明确告知"我需要查询确认"
3. 复杂问题主动引导至人工客服
4. 禁止编造不存在的产品信息或政策

1.3 RAG：外部知识检索增强

RAG（Retrieval-Augmented Generation）通过向量检索将外部知识库与 LLM 结合，解决知识边界与实时性问题。

RAG 的技术价值
  • 知识库可动态扩展，无需重新训练模型
  • 文档更新即时生效，保证信息时效性
  • 回答可追溯至原始文档，降低幻觉风险
  • 实现成本显著低于模型微调

1.4 Context Management：对话状态维护

Context Management 通过维护对话历史与关键状态，解决多轮对话中的信息丢失问题。

关键认知

经过 Prompt Engineering、RAG 和 Context Management 三层增强，系统已具备"合格客服"的能力：理解业务语境、检索专业知识、维护对话连续性。但此时系统仍属于被动响应型，只能"回答问题"，无法"执行操作"。

从 LLM 到 Agent 的分水岭，在于系统是否具备主动执行能力。

二、Agent 架构的核心组件

2.1 Function Calling：工具调用机制

Function Calling 使 LLM 能够识别需要外部工具支持的场景，并生成结构化的工具调用请求。

tool-calling.json 
// 用户输入：帮我查一下订单 12345 的物流

// LLM 判断需要调用查询工具
{
  "tool": "get_logistics",
  "parameters": {
    "order_id": "12345"
  }
}

// 系统执行工具调用，返回结果
{
  "status": "shipped",
  "carrier": "SF",
  "tracking_no": "SF1234567890",
  "eta": "2024-03-15"
}

// LLM 基于工具返回生成回复
"您的订单已发货，顺丰快递单号 SF1234567890，预计 3月15日送达。"
维度 LLM（增强后） Agent
交互模式 被动响应 主动执行
能力边界 文本生成 文本生成 + 工具调用
系统角色 顾问（Advisor） 执行者（Executor）
典型应用 知识问答 订单处理、退款申请

2.2 ReAct：推理-行动循环框架

ReAct（Reasoning + Acting）是 Agent 系统的核心编排模式，通过显式的推理-行动循环实现复杂任务处理。

react-loop.md 
ReAct 循环模式：

1. Thought（推理）
   分析当前状态，制定下一步行动计划

2. Action（行动）
   调用具体工具执行操作

3. Observation（观察）
   获取工具执行结果

4. 循环
   基于观察结果继续推理，直至任务完成

ReAct 的技术优势：

  • 可解释性：每一步决策过程透明可追溯
  • 灵活性：支持基于中间结果的动态调整
  • 可调试性：故障定位可精确到具体步骤

2.3 Memory：分层记忆机制

特性 短期记忆（Short-term） 长期记忆（Long-term）
数据范围 当前会话上下文 跨会话用户画像
存储周期 会话级（Session） 持久化（Persistent）
典型内容 当前订单号、对话轮次 用户偏好、历史行为
技术实现 Redis / 内存缓存 向量数据库 / 图数据库

2.4 Agent 系统架构

Model Layer（认知层）
GPT-4 / Claude / Gemini
职责：语义理解、推理决策、文本生成
↑ ↓
Orchestration Layer（编排层）
• Planning（任务规划）
• Memory Management（记忆管理）
• Context Engineering（上下文组装）
• ReAct Loop（推理-行动循环）
↑ ↓
Tool Layer（工具层）
• 订单查询 API
• 退款申请 API
• RAG 检索服务
• 外部系统接口
架构核心洞察

Agent 系统的本质是将 LLM 的认知能力与外部工具的执行能力通过编排层有机整合，实现从"能说"到"能做"的能力跃迁。

三、能力成熟度模型

Google 提出的 Agent 五级能力模型，为技术演进提供了清晰的阶段划分：

Level 名称 技术特征 典型能力
L0 核心推理系统 纯 LLM，无外部交互 基于训练知识的文本生成
L1 连接型问题解决者 + Tool Use 实时数据查询（搜索、API）
L2 战略型问题解决者 + Planning + Context 多步骤任务规划与执行
L3 协作型多智能体系统 Multi-Agent 架构 专业化 Agent 分工协作
L4 自我进化系统 自主能力扩展 动态创建工具与 Agent（研究中）

四、产品演进路线图

基于技术成熟度与业务价值的平衡，建议采用以下五阶段演进策略：

阶段一：MVP（最小可行产品）

  • 技术栈：LLM + Prompt Engineering
  • 核心能力：标准 FAQ 回答
  • 投入成本：低
  • 阶段目标：验证需求，快速上线

阶段二：知识增强

  • 技术栈：+ RAG
  • 核心能力：企业私有知识问答
  • 投入成本：中
  • 阶段目标：提升回答准确性与时效性

阶段三：功能扩展

  • 技术栈：+ Function Calling
  • 核心能力：订单查询、状态变更等操作
  • 投入成本：中高
  • 阶段目标：从"能说"到"能做"

阶段四：智能升级

  • 技术栈：+ ReAct + Memory
  • 核心能力：复杂任务自主规划与执行
  • 投入成本：高
  • 阶段目标：高价值场景的深度自动化

阶段五：规模化

  • 技术栈：+ Multi-Agent + 长期记忆
  • 核心能力：个性化服务、跨系统协作
  • 投入成本：很高
  • 阶段目标：企业级智能服务中台

结论

从 LLM 到 Agent 的技术演进，本质是从静态知识到动态能力的转化过程。作为产品经理，在技术选型时应始终回归业务本质：

  • 用户的真实痛点是什么？
  • 最小可行的技术方案是什么？
  • 技术投入的 ROI 如何评估？

技术概念的迭代速度远超业务场景的演变，理解底层逻辑比追逐最新技术更为重要。建立清晰的技术认知框架，才能在快速变化的技术浪潮中做出稳健的产品决策。

本文通过智能客服这一具体场景，梳理了 LLM、RAG、Agent 等核心概念的逻辑关系与技术边界。希望读者能够基于此框架，在实际产品工作中做出适合自身业务的技术选型。

术语表

  • LLM（Large Language Model）：大语言模型，基于 Transformer 架构的预训练语言模型
  • Prompt Engineering：提示工程，通过结构化指令优化模型输出的技术
  • RAG（Retrieval-Augmented Generation）：检索增强生成，结合外部知识检索的文本生成方法
  • Function Calling：函数调用，LLM 生成结构化请求调用外部工具的能力
  • Agent：智能体，具备感知、决策、执行能力的自主系统
  • ReAct（Reasoning + Acting）：推理-行动框架，显式化推理过程的 Agent 编排模式
  • Context Management：上下文管理，维护对话状态与历史信息的技术
