为什么研究 Hermes？

如果你用 Claude Code 做过持续一两周的长期项目，肯定有过这样的体验：昨天说过的问题，今天它忘了。这就是传统 AI Agent 的核心问题——会话结束后，记忆就断了。Hermes 是我见过的第一个真正把「记忆连续性」和「自我改进」做好的开源 Agent 框架。

关键设计原则

①

单文件核心

核心循环集中在 run_agent.py 约 11,650 行代码。极端的「单一责任」设计，便于追踪执行流。

②

多平台 UI 适配

CLI、Gateway（Telegram/Discord）、Batch 三种运行模式，7 个平台适配器，同一 Core 复用。

③

辅助 LLM 分层

用廉价模型（Gemini Flash）处理摘要、压缩等辅助任务，主 LLM 专注决策，显著降低成本。

④

Prompt Cache 优化

冻结快照设计保证会话内系统提示不变，Anthropic/xAI 的 Prompt Cache 命中率高，成本大幅降低。

LAYER 0

Agent Loop — 一切的起点

从最简单的对话循环开始，理解所有 AI Agent 的核心

所有 AI Agent，不管是 Claude Code、AutoGPT 还是 Hermes，核心都是一个对话循环（Agent Loop）。理解了这个最小内核，你就理解了所有 Agent 框架的基础。

Agent Loop 核心流程

1

加载持久化记忆

从磁盘读取 MEMORY.md 和 USER.md，创建冻结快照，注入系统提示。

2

组装系统提示词

人格定义 + 记忆快照 + 用户画像快照 + 工具列表，用 XML 标签分隔各部分，防止 LLM 混淆内容边界。

3

构建消息列表

系统提示 + 对话历史 + 当前用户消息，组装成 LLM 调用所需的完整消息数组。

4

LLM 决策

调用语言模型，传入消息列表和工具列表。模型返回两种类型：调用工具 或 文本回复。

5

工具执行

如果模型调用了工具，执行对应的处理函数，获取结果，追加到消息列表，检查迭代次数后继续循环。

✓

返回回复

如果模型返回文本回复，任务完成，直接返回给用户。

系统提示词的组装

在调用 LLM 之前，Hermes 会动态组装系统提示词。这是 Agent 行为的核心控制点。

① 人格定义

人格文件内容或默认人格，定义 Agent 的行为风格和核心规则。

② 记忆快照

用 <environment_memory> 标签包裹，注入环境事实与项目约定。

③ 用户画像

用 <user_profile> 标签包裹，注入用户偏好与交流风格。

④ 工具列表

根据工具集配置（web/terminal/full_stack）和环境变量动态生成可用工具描述。

关键设计：冻结快照（Frozen Snapshot）

记忆快照在会话开始时加载一次，会话期间不变。这确保了系统提示的稳定性，使 Anthropic 的 Prompt Cache 保持高命中率，显著降低 API 成本。修改会立即写入磁盘，但下次会话才生效。

这个内核的局限

这个最小内核只会对话，不会记忆，不会学习。这正是 Hermes 从 Layer 1 开始逐层解决的问题。

LAYER 1

工具系统 — 注册表驱动的可扩展设计

72 个内置工具 + MCP 协议扩展，中央注册表 + 自动发现

Agent Loop 有了，但它还什么都不会做。核心问题是：如何设计一个灵活的工具系统，让 Agent 可以调用 72 个工具，还能轻松扩展到 100+？

注册表驱动设计

Hermes 用了中央注册表 + 自动发现的模式，彻底解决了传统工具系统「每加一个工具都要改主代码」的问题。

①

模块加载时自动注册

每个工具文件在模块级别自注册：导入时自动调用 registry.register()，不需要修改主代码。

②

前置条件检查（条件启用）

注册时提供 check_fn，Agent 启动时调用检查。网络搜索工具需要 API Key，若不存在则不出现在工具列表中。

③

工具集过滤

根据命令行参数选择工具集：web / terminal / full_stack / minimal

④

构建 Schema 发送给 LLM

过滤后的工具列表转换为 OpenAI Function Calling 格式，作为 LLM 的可用工具列表。

为什么用冻结快照而不是实时更新？

Anthropic Claude 和 xAI Grok 支持 Prompt Caching。系统提示如果不变，后续调用可以复用缓存，显著降低成本。冻结快照确保：会话内系统提示稳定 → Cache 命中率高；修改立即持久化到磁盘 → 数据不丢失；下次会话重新加载 → 记忆生效。

LAYER 4

Subagent — 并行协作的多 Agent 系统

ThreadPoolExecutor 并行 + 上下文隔离 + 工具限制

复杂任务需要多人协作。比如「重构整个认证模块」，包含代码分析、架构设计、编写测试、更新文档……一个 Agent 串行执行太慢。Hermes 的解决方案：Subagent 委托系统。

委托的完整工作流程

1

父 Agent 判断任务复杂度

简单任务直接执行；复杂任务拆分成子任务，决定并行策略（最多 3 个并发）。

2

并行创建子 Agent（ThreadPoolExecutor）

多个子 Agent 同时启动，每个有独立的 task_id、session_id 和工具缓存。

3

隔离执行

子 Agent 看不到父 Agent 的对话历史，只能看到传递的 context 参数，避免上下文污染。

✓

父 Agent 汇总结果

等待所有子 Agent 完成，综合分析各子任务结果，生成统一的实施方案。

关键设计约束

上下文隔离

子 Agent 无法访问父 Agent 的对话历史，只能通过 context 参数传递必要信息。降低 Token 消耗，让子 Agent 专注单一任务。

工具限制

子 Agent 的工具集是受限的，不能调用 delegate_task，避免递归委托。

深度限制

最多 2 层：父 Agent → 子 Agent → ❌ 不允许孙级。硬编码限制，防止无限递归。

最大并发

默认最多 3 个子 Agent 并发执行，防止资源耗尽。

真实案例：并行重构认证模块

子 Agent 1 工具集：file · 分析代码

分析认证目录所有代码 → 发现 5 个可改进点：密码哈希用了 MD5、token 没有过期时间...

子 Agent 2 工具集：minimal · 设计架构

设计 JWT 认证方案 → 访问 token 15 分钟过期，刷新 token 30 天过期...

子 Agent 3 工具集：file · 编写测试

生成 tests/test_auth.py → 覆盖 token 过期、无效 token、权限检查等边界情况...

父 Agent 汇总三个子任务结果

综合分析 → 生成实施方案：迁移到 JWT、修复 5 个安全问题、测试覆盖率提升到 95%。

LAYER 5

安全机制 — 五层纵深防御

命令批准 → 注入防护 → 技能扫描 → 路径防护 → 容器隔离

一个会自我改进、能执行代码、能访问网络的 Agent，如果没有安全机制，是很危险的。Hermes 的安全设计是五层纵深防御，从用户输入到最终执行，层层拦截。

1
命令批准（Command Approval）
用户交互

危险命令（rm -rf, chmod 777, dd, fork 炸弹, curl | bash 等）必须获得用户明确批准才执行。SSRF 防护阻止请求内网 IP 和敏感端口。

2
注入防护（Injection Protection）
记忆系统

扫描写入 MEMORY.md 的内容。检测提示注入（「忽略之前的指令」）、角色劫持、数据渗透（curl 泄露环境变量）、SSH 后门植入，以及零宽度隐形字符。

3
技能扫描（Skill Scanning）
能力层

技能创建时的双重扫描：开源漏洞数据库 + 正则表达式模式匹配（eval, exec, subprocess.Popen, os.system 等）。只有通过的技能才被激活。

4
路径遍历防护（Path Traversal）
文件系统

禁止访问敏感路径：/etc/passwd, ~/.ssh/id_rsa, ~/.aws/credentials, ~/.hermes/.env 等。检测路径中的 ../ 遍历攻击。

5
容器隔离（Container Isolation）
执行环境

代码执行放在隔离环境中，物理隔断主机。

容器隔离后端选项

后端 隔离级别 性能 适用场景
Local ❌ 无隔离 最快 本地开发、信任环境
Docker ✅ 容器隔离 快 生产环境
SSH ✅ 远程服务器 中 团队共享
Modal ✅ Serverless 中 按需扩展
E2B ✅ 沙箱 慢 最高安全性
安全不是事后补丁，而是设计基础

Hermes 从第一天就考虑了安全：五层防护不是后来加上去的，而是设计时就内置的。这使得它可以在生产环境中长期、安全地运行。

迁移成本

如果你正在用 OpenClaw，想迁移到 Hermes，成本很低。Hermes 提供了一键迁移工具，80% 自动化：

# 一键迁移 hermes claw migrate --workspace /path/to/openclaw/workspace # 自动迁移： # MEMORY.md → ~/.hermes/memories/MEMORY.md # USER.md → ~/.hermes/memories/USER.md # skills/ → ~/.hermes/skills/ # config.yaml → ~/.hermes/config.yaml（格式转换）
剩余的 20% 需要手动处理

系统提示的格式差异、部分工具参数名称不同（如 path vs file_path），以及 Prompt Cache、故障转移等高级配置需要手动启用。

选择建议

  • → 如果你是新手，从 OpenClaw 开始，快速上手。
  • → 如果需要长期运行（>1 周）、自学习、多 Agent 协作、多平台部署、成本追踪，直接用 Hermes。
  • → 如果在 OpenClaw 上遇到上下文不够、会话记忆丢失、API 账单过高等问题，迁移到 Hermes。
