Workflow vs Agent：什么时候该用什么

开篇

Workflow 算不算 Agent？Agent 能自主规划是不是就更先进？多 Agent 系统是不是更高级？

很多人陷进了一个误区——复杂度崇拜。觉得越复杂的系统越厉害，越多的 Agent 越智能。

但 Anthropic 的这篇《Building Effective AI Agents》讲得非常清楚：Workflow 和 Agent，本质上都属于智能体系统。真正的区别，不在有没有大模型——而在决策权归谁。

复杂度崇拜的误区

在 Agent 热潮中，很多人都有一个隐含的假设：Agent 比 Workflow 高级，Workflow 只是 Agent 的低配版。

但这种认知是错误的。

核心观点

Workflow 不是低级方案，Agent 也不是高级标签。不管黑猫白猫，能抓老鼠的就是好猫。

核心区别：决策权归谁

Workflow 和 Agent 的本质区别，在于谁拥有决策权。

Workflow Agent
路径 人写死的 模型决定的
步骤 提前设计好的 可以动态调整
模型角色 执行者 决策者
特点 可控、可复现、好调试 自主性强，但不确定性也来了

Workflow：路径是人写死的，步骤是提前设计好的，模型只是执行。决策权在人手里。它可控、可复现、好调试。

Agent：路径由模型决定，过程可以动态调整。它会规划、会选择工具、会根据结果改变策略。决策权在模型手里。但自主性一上来，不确定性也上来了。

关键问题

同样的问题，今天一个答案，明天一个逻辑。企业怎么用？业务怎么验收？

什么时候该用什么？

所以关键问题是：什么时候该用 Workflow？什么时候才需要 Agent？

我总结了三个判断标准：

1

路径明确、步骤固定 → 流程问题

如果任务有明确的执行路径，每一步都是固定的，那就是流程问题，用 Workflow。

2

目标稳定、可枚举 → 流程问题

如果目标是稳定的、清晰的、可枚举的，不需要动态策略，就不要引入 Planning。

3

以上都不是 → 才考虑 Agent

只有当任务路径无法通过硬编码实现，或者需要多轮操作时，才轮到 Agent。

核心原则

不要把流程问题做成模型问题。

增强型 LLM

什么是增强型的 LLM？

提示词写清楚，能检索，能调用基础工具，能记住上下文。很多场景，到这一步已经够了。

增强型 LLM：基础构建块，具备检索、工具和记忆能力

实现这些增强功能的关键在于两个方面：根据你的具体用例定制这些能力，确保它们为 LLM 提供简单、文档完善的接口。

Anthropic 最近发布的 Model Context Protocol (MCP) 就是一种实现方式，它允许开发者通过简单的客户端实现与不断增长的第三方工具生态系统集成。

Workflow 详解

当任务有明确步骤时用 Workflow。根据任务特点，有 5 种常见的 Workflow 模式：

1. Prompt Chaining（提示链）

如果任务可以被拆成顺序执行的子任务——那就用提示链。每个 LLM 调用处理前一个调用的输出。

提示链工作流：将任务分解为顺序步骤

应用示例：

  • 写营销方案：先出大纲，再写内容
  • 生成文案，然后翻译成不同语言

2. Routing（路由）

如果任务有多个处理路径，那就用路由。对输入进行分类，并引导至专门的后续任务。

路由工作流：将输入分类并引导至专门处理流程

应用示例：

  • 智能客服：根据用户意图不同，是售前咨询、售后反馈，进入下游不同的路径
  • 简单问题用小模型，困难问题用大模型

3. Parallelization（并行化）

如果子任务彼此独立，或者希望从多个视角分析问题，提高结果的置信度，那就并行。一起跑，最后再汇总。

并行化工作流：同时处理任务并聚合结果

应用示例：

  • 审查代码漏洞，多个提示审查代码并在发现问题时标记
  • 评估内容是否不当，多个提示评估不同方面

4. Orchestrator-Workers（编排器-工作者）

如果任务变复杂，没办法提前写好所有子步骤，那就需要一个协调器。由一个中央模型动态拆解任务，分配给不同节点，再整合结果。

编排器-工作者工作流：中心节点动态分解和委派任务

应用示例：

  • AI Coding：改哪些文件，要根据具体需求现场决定
  • 搜索任务，涉及从多个来源收集和分析信息

5. Evaluator-Optimizer（评估器-优化器）

如果任务需要反复优化，那就加一个评估优化器。一个模型负责生成，一个模型负责评估。

评估器-优化器工作流：迭代改进响应质量

应用示例：

  • 做 Deep Research：要要不要继续检索？信息够不够？需要一个评估节点来判断
  • 文学翻译，评估器可以提供有用的批评

Agent

当出现探索型任务，任务路径无法通过硬编码实现，或者需要多轮操作时，才引入 Agent。

但是要注意：

  • 在执行过程中，要有环境数据反馈，比如工具的调用结果、上一步执行的结果等
  • 敏感问题需要引入检查点，引入人类反馈来监督执行
  • 设置最大迭代次数，避免系统无限循环

自主 Agent：基于环境反馈循环使用工具

注意事项

Agent 的自主性意味着更高的成本和潜在的复合错误。建议在沙盒环境中进行广泛测试，并设置适当的防护栏。

总结

Workflow 不是低级方案。Agent 也不是高级标签。不管黑猫白猫，能抓老鼠的就是好猫。

复杂系统，不是设计出来的。是从简单系统一步一步迭代出来的。

真正拉开差距的——不是你堆了多少 Agent。而是你能不能设计好决策权的边界。
