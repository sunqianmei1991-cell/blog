如果你正在做多 Agent，或者还在犹豫要不要做，我建议你先停下手里的活，花几分钟听听 Anthropic 刚发的这篇干货。他们最新研究显示：多 Agent 系统比单 Agent 性能高出 90.2%。

你让 AI 调研“2025 年全球 AI 芯片市场”。你丢给他 200份财报、50 篇行业研报，会发生什么





注意力稀释：他读到第 150 份时，早忘了前 10 份的核心数据了。即使是天才也会淹没在海量的数据里



串行效率极低：他只有一双手，搜一个、看一个、读完再搜下一个。等他看完，2026 年都快结束了。



路径漂移：他又要干活又要做计划。搜着搜着就钻到的细节里出不来了，完全忘了你要做什么。

Anthropic 发现，单纯堆模型能力没用，真正的解法是：把一个人变成一支特种部队，Anthropic 将其抽象为Orchestrator-Subagent架构

Orchestrator（老板）：大脑，负责战略决策

老板不亲自去读网页、翻文档。它的工作是制定计划、统筹全局、分配任务，并且把计划写在本子上随时查阅。

Sub-agents（员工）：手脚，负责具体执行

每个员工只专注一个极其具体的任务：





“有的负责提取财务数据”



“有的负责搜索政策”



“有的负责对比产品”

他们不需要操心全局，只需要把自己那块工作做好、做快、做准。



这套架构最精妙的地方在于：它运行的不是预先写好的固定流程，而是 OODA 循环

Observe (观察)汇总已有的数据和员工反馈

Orient (定位)分析现状，判断离目标还差什么

Decide (决策)决定是继续深挖还是换个路子

Act (执行)下发指令给子 Agent 调工具



在 Anthropic 内部的调研系统中，一次典型的运行是这样的：





创建 LeadResearcher 实例，它首先生成一个“初始研究大纲”。



 它会将计划存入外部存储（Memory）。



 根据大纲，瞬间创建 多个subagent去全网搜索。



每个subagent在返回结果前，必须自我反思：我搜到的东西对老板有用吗？



最后由 LeadResearcher 汇总所有碎片，交给最后的 CitationAgent。这个角色非常重要，它不写内容，只负责检查：报告里的每一句话，到底能不能在原文里找到证据



多agent系统如此复杂，Anthropic 在建设这套系统时遇到了不少挑战。

实验室版本和工业级版本往往就差在“最后一公里”上

上下文截断导致“系统性失忆”

由于每个subagent信息都汇总给 Lead Researcher，海量数据会导致对话历史迅速突破 20 万 Token 限制。一旦发生截断，Lead 就会失忆。

解法是：增加一个外部 Memory：不依赖原生上下文，将核心计划存入外部存储，保证全局目标不随 Token 滚动而丢失

工具描述不清，AI“瞎用”

工具调用失败往往不是模型弱，而是人写的说明书存在歧义，导致 AI 误解了工具的用法

解法是：





AI 写说明书：开发 Tool-testing Agent，让它多次使用失败的工具，在失败中摸索工具的坑和逻辑漏洞，重写工具说明书。仅此一项，任务耗时就能降低 40%。



交错思考：提示词中强制模型在调用工具前后进行显式思考，调用前说明理由和预期，调用后评反馈。

路径随机导致“评估失灵”

多 Agent 是非线性的，同样的查询跑两次路径都不同，传统的“准确率”根本没法衡量系统好坏

解法是：





模型辅助评估：引入高阶模型作为“裁判”，针对报告的事实性和引用准确性进行逻辑打分。



小样本快跑：选取 20 个最具代表性的真实案例测试。在早期，这几十个案例就能帮你揪出架构中 80% 的系统性缺陷

成本膨胀，任务规模难控制

多 Agent 系统可能为了简单问题盲目开启几十个 subAgent，导致 token 消耗呈指数级增长。

解法： 在提示词中嵌入规模自适应规则：





简单事实查询：1 个 subAgent，3-10 次工具调用



直接对比任务：2-4 个 subAgent，各 10-15 次调用



复杂研究：10+个 subAgent，明确划分职责

信息碎片化导致“张冠李戴”

多 Agent 并行处理后，信息极其碎片化，容易出现“结论与原始来源无法对应”的问题。

解法： 前文提到的 CitationAgent（对账员），专职负责引用核查，确保所有主张都有据可查



这套架构的核心思想很简单：





不要期待一个超级大脑做所有事



让专业的人做专业的事



用协作代替单打独斗

就像公司老板不可能一个人做完所有事，必须靠团队协作。AI 智能体的未来，也一定属于那些懂得“分工协作”的系统。
